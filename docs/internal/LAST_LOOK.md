# Spectral — the "last look" before dismissal (2026-07-03)

Before dismissing Spectral as not-better-than-the-boring-stack for Permagent, we
ran an adversarial multi-agent sweep of the whole codebase (27 agents) looking
for missed wiring, a simple change, an unclosed loop, or an untested capability —
then closed the one loop that survived scrutiny.

## Swarm verdict: dismissal warranted, one loop worth closing

Of 20 candidate findings, 3 survived adversarial verification. The synthesis:
**nothing found flips "use the boring stack" for Permagent**, but co-retrieval
was the one never-measured capability that is both genuinely unique to Spectral
(cross-query co-access memory — pgvector/MinHash+BM25 do pure per-query
similarity) and cheap to finally test.

### Confirmed dead on inspection (do not relitigate)
- **Update-handling / T4 (Zep's turf):** killed on data — all 106
  `consolidation_edges` have byte-identical source/target content. It's a dedup
  ledger, not supersession. No `valid_to`/MERGE/supersede logic; no stale-vs-
  current signal exists in this brain.
- **Wing-precision / T3:** tautological — `rc_focus_wing` is consumer-*supplied*,
  so a plain `WHERE wing = focus` filter hits ~100% by construction.
- **Federation:** no moat — `fan_out_recall` is a sequential loop over the Brain
  API; any N-store loop over pgvector/MinHash reproduces it.
- **Fingerprints-into-ambient-ingest:** real code gap, wrong causal story —
  co-retrieval rides `co_retrieval_pairs` (from `retrieval_events`), not
  `constellation_fingerprints`; wiring fingerprints in changes nothing.
- **Recognition/spectrogram salvage, compaction tiers, dead `signal_scorer.rs`:**
  all real observations, all nicer-but-inert. None touch the decision.

### Methodology fact worth recording
`recall()` / `recall_at()` **bypass the cascade re-ranking pipeline**, and
`spectral-bench-real` calls `recall()`. So ambient boost, co-retrieval, density,
diversity, and dedup have never been exercised in the realistic bench setting —
the differentiated retrieval stack was never actually on trial. (Doesn't rescue
the thesis; the signals are expected inert at this volume — but co-retrieval, we
now know, is the exception.)

## The co-retrieval ablation (the closed loop)

Harness: `crates/spectral-bench-real/src/bin/coret_ablation.rs`.
Data: real brain snapshot (`~/.permagent/brain`), 144,542 co-access pairs rebuilt
from 744 `retrieval_events`. 300 content-derived proxy queries, FTS pool 40,
weight 0.10 vs an empty boost map. Episode-diversity + context-dedup OFF so the
only difference between arms is the co-retrieval weight — a pure permutation diff.
Raw: `~/spectral-local-bench/coret-ablation.json`.

**Scope, stated honestly:** this measures whether the lever FIRES and how much it
reorders — **not** whether the reordering is *better*. It cannot: the co-access
pairs are generated by Spectral's own cascade, so any relevance label mined from
them (`recognition_set_members`) is circular, and this brain has no independent
label. Movement is the only legitimate label-free readout.

### Result — it fires, hard (the opposite of the spectrogram's 0/500)

| metric | value |
|---|---|
| queries with a co-access boost surface | 99.3% |
| mean boosted candidates per 40-pool | 18.2 |
| **queries with a changed top-5 *set*** | **54.7%** |
| queries with changed top-10 order | 93.3% |
| mean \|Δrank\| / max | 0.73 / 9 |

Co-retrieval is a live, high-impact reranking signal on the real brain — it
changes the top-5 membership for the **majority** of queries, and it is already
ON in production at weight 0.10.

## What this means (no hope manufactured)

1. **The loop is closed on "does it fire": emphatically yes.** This is the one
   place Spectral demonstrably does something the boring stack can't, at scale.
   It is not dead weight.
2. **It just relocates the open question to "is the firing good?"** — which is
   unmeasured and, on current data, unmeasurable. A signal that changes half the
   top-5s while unvalidated is as much a **production risk** as an opportunity.
3. **The decision vs the boring stack does not change today.** You still cannot
   claim a recall win. But "co-retrieval: never measured" is now "co-retrieval:
   fires hard, unvalidated, with a defined path to resolution."

## The single next step that would actually resolve it

The blocker is a **non-circular relevance label on the real workload**. Two facts
govern the path:
- An LLM relevance judge is *not* circular (it rates query↔memory relevance
  independently of the co-access graph) — so a label is obtainable.
- But real query **text** isn't retained (`retrieval_events` stores only
  `query_hash`). Today we can only judge *content-derived proxy* queries, not the
  real ones.

So the true unlock is a **Permagent-side data change: log the query text** (or an
A/B outcome signal) going forward. With that, a tightly-scoped judge pass — only
the ~55% of queries whose top-5 changed, arm A vs arm B, blind — settles whether
co-retrieval helps, hurts, or is noise, for a small spend.

Optional cheap directional check now: an LLM-judge pass over the proxy queries
would hint at sign, but it is not the real workload and would not be decisive.

**Bottom line:** Spectral remains, today, not-better-than-the-boring-stack for
Permagent — the dismissal stands. But the last look found one real, differentiated,
high-impact thread that is currently running **unvalidated in production**, and it
converts a vague "never measured" into a crisp, cheap, well-defined experiment
gated only on retaining query text. That is the one thing worth doing before —
or instead of — shelving it.
