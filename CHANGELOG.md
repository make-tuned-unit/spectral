## [Unreleased]

### Changed (feat/porter-default)
- **Porter stemming is now the default FTS tokenizer** (`"porter unicode61"`). Plural/inflected queries bridge to singular content ("doctors" → "doctor") deterministically, at zero runtime cost — no LLM, no embeddings. Validated Tier-0 (zero-evidence questions 8→4, answer-key recall 51.8%→54.7% on LongMemEval-S) and Tier-1 (deterministic porter-only pipeline at parity with LLM-assisted query expansion, 46/60 vs 43/60). Configure via `BrainConfig::fts_tokenizer`, the `BrainBuilder::fts_tokenizer()` builder method, `SqliteStoreConfig::fts_tokenizer`, or the `SPECTRAL_FTS_TOKENIZER` env var; set `"unicode61"` (or empty) to restore unstemmed matching.
- **Automatic FTS tokenizer migration**: opening a brain whose FTS index was built with a different tokenizer triggers a one-time index rebuild (drop + recreate + repopulate; the memories table is untouched). Existing brains pick up porter stemming on next open with no re-ingest.
- **Re-rank pool widening is now a library knob**: `RecallTopKConfig::fetch_mult` (default 3) fetches `k × fetch_mult` FTS candidates as the re-ranking pool and truncates to `k` after re-ranking. Recovers evidence that broad stemmed matching pushes below the fixed bm25 LIMIT (+0.8pp temporal session recall at mult=3, ~+4% context tokens; validated in docs/internal/TIER1_PORTER_WIDEN.md). Previously bench-only via `SPECTRAL_TOPK_FETCH_MULT`; the env var remains as an ablation override in `spectral-bench-accuracy`. Set `fetch_mult: 1` to disable.

### Added (feat/fts-stopwords)
- **Optional FTS stopword filtering** (`SPECTRAL_FTS_STOPWORDS=1`, off by default). The second issue the possessive-recall investigation exposed: query terms like "is"/"what"/"the" match unrelated turns (a turn whose only overlap with the query is "is" enters the candidate pool as noise, and under a tight K can crowd out real matches). The filter uses a **conservative, homograph-safe** function-word set — deliberately excluding ambiguous tokens that are often content in personal memory (`it`/`IT`, `us`/`US`, `can`, `may`, `will`, `march`, `in`, `on`, `at`) — with a fallback that never empties an all-function-word query. Flag-gated (measure on the real bench before defaulting, per the porter/co-retrieval discipline). Verified deterministically: drops the "is"-only noise turn while keeping the answer turn; unit-tested for possessive stripping, stopword drop, empty-query guard, and homograph safety.

### Added (bench/scale-validation)
- **Recognition validated at scale** (`spectral-bench-real --bin recognition_scale_bench`). The 12-item recognition bench proved the mechanism; this enrolls **300** templated incidents with overlapping domain vocabulary and stresses the real scale risk — false positives from chance overlap. With the shipped defaults (`shingle=2`, `min_similarity=0.15`): **ROC-AUC 0.996, exact re-encounter 1.00, degraded@72% 0.98, near-miss novel 1.00, unrelated novel 1.00** (zero false positives on either negative class). A config sweep **confirms the defaults**: `shingle=3` is strictly worse (AUC 0.996→0.946, degraded recall 0.98→0.78, because heavy degradation breaks 3-grams more than 2-grams) and `min_similarity` is insensitive across [0.15, 0.35]. The bench also documents the recognizer's contract — it is a *lexical* near-duplicate detector: it correctly calls textual near-duplicates familiar and differently-phrased domain-related text novel; distinguishing "same phrasing, different entity" is out of lexical scope by design. **Design cross-checked against the near-duplicate literature** (Broder shingling; Charikar SimHash; MMDS LSH; Koslicki & Zabeti containment-MinHash; SpotSigs): the two highest-value levers it recommends are already met or bettered — recognition scores **exact** shingle-set containment (`minhash::containment`, `lib.rs:205`), which is the unbiased ground truth that the recommended Bloom-filter containment *estimator* only approximates; and candidate generation uses an **exact inverted shingle index**, which the literature notes strictly dominates LSH banding on recall. SimHash is the wrong tool here (symmetric cosine, no containment — it can't score a fragment-in-document match). The one genuinely additive idea (SpotSigs stopword-anchored signatures) targets boilerplate-heavy web documents, a poor fit for short agent-memory items; logged as a future option, not adopted.
- **RRF-k robustness confirmed** (`--bin fts_fusion_experiment`): fused recall@1 is **1.00 across k ∈ [1, 300]** for the porter+unstemmed channel pair, so the field-default k=60 is a sound, non-fragile choice — no tuning burden.
- **Ambient feedback (lift anticipatory recall) validated at scale** (`--bin ambient_scale_bench`). 12 topics × 8 memories + one globally-popular memory, driven to a **1287-pair** co-retrieval graph with a diffuse popularity trap (each seed's raw co_count with the popular memory exceeds its co_count with its own topic siblings). Aggregated over all topic seeds: **precision@5 (same-topic) — lift 1.00 vs raw co_count 0.38**, and lift **never** surfaces the globally-popular memory (co_count's recommendations are 62% wrong-topic, dominated by the cross-topic/popular co-occurrences whose raw counts tie the genuine siblings). Lift maintains perfect topic targeting and fully suppresses the popularity trap at scale — confirming the `recommend`-by-lift design over `related_memories`-by-co_count.

### Added (feat/fts-fusion)
- **Stemmed + unstemmed RRF fusion for FTS recall** (`SPECTRAL_FTS_FUSION=1`, off by default). Porter stemming (the shipped default) is a recall device that trades away precision: it bridges `doctors`→`doctor` (a recall win) but also conflates `university`→`univers`←`universe`, so a short colliding distractor can outrank the answer at a tight `k`. With fusion on, the store maintains a second, **content-only, unstemmed (`unicode61`)** FTS index (`memories_fts_raw`, kept in sync by triggers, purged on delete by the AFTER DELETE trigger — no change to the forget path) and, at query time, fuses the porter-ranked and unstemmed-ranked lists by **Reciprocal Rank Fusion** (Cormack 2009, k=60). The unstemmed channel applies a **conservative regular-plural strip** at query time (`engineers` → `"engineers" OR "engineer"`; single trailing `s` only, skipping `ss`/`us`/`is` and len≤3 words), making it S-stemmer-like: it bridges regular plural↔singular that pure-unstemmed would miss, *without* porter's aggressive over-stemming — so it recovers a rarer inflected answer (`engineer`) that a common-form flood (`engineering`→`engin`) buries under porter, while leaving the collision fixes (`university`, `policy`, …, none of which end in `-s`) untouched. Different *representations* (the Beitzel precondition where fusion actually helps) recover both failure modes; rank-based fusion means **no score normalization** across the two BM25 scales. Fusion is a store-construction property, so `recall_topk_fts` is unchanged — it transparently fuses. **Measured (fusion micro-bench, 18-memory corpus with both inflection and over-stemming cases): recall@1 0.57 (porter) / 0.57 (unstemmed) / 1.00 (fused)** — fused dominates both channels and never underperforms either. Validated end-to-end through the `Brain` (`brain_tests::fts_fusion_recovers_overstem_and_inflection_end_to_end`) and for deletion safety (`forget_purges_fusion_raw_index_no_resurrection` — a forgotten memory cannot resurrect via the unstemmed channel). Opt-in per the measure-before-defaulting discipline (it costs a second FTS index); a read-only replica fuses only if the raw index was already synced into the file. Ceiling measured in `spectral-bench-real --bin fts_fusion_experiment`; the null result for BM25 `key`-column reweighting is in `--bin bm25_weights_experiment`. **Scale validation (`--bin fusion_scale_bench`, 330-memory corpus, neutral-dominant query mix, opaque content-hash keys, full recall pipeline):** ALL-query **recall@1 0.70 → 1.00 (+0.30)**, collision bucket **0.14 → 1.00 (+0.86)**, inflection **0.83 → 1.00 (+0.17, the plural-strip win)**, neutral **1.00 → 1.00** (no regression) — **every answer rank 1 under fusion, zero queries where fusion lost an answer porter retrieved**; the collision win concentrates at tight k (by recall@10 porter catches up), matching the precision-at-tight-k prediction. **Critical operating-point caveat:** at **recall@40 — LongMemEval's actual retrieval K — the porter-vs-fusion delta is +0.00** on this corpus (both 1.00). Fusion's benefit here is tight-k *reordering* (rank 32 → rank 1), not recall *expansion*; when the actor reads all K=40, an answer that porter buries but still keeps within the top-40 is already visible, so fusion changes nothing it sees. Fusion can only move a large-K benchmark when an over-stemming flood pushes the answer OUT of the fetch pool (`k × fetch_mult` = 120) entirely and the unstemmed channel pulls it back — which needs haystacks far larger than this synthetic corpus. On LongMemEval that condition is **unverified**; do not assume fusion lifts the large-K number without the real-corpus run. **Latency cost (the "least expensive" half):** the second FTS query + id-fetch adds **+0.45 ms/query median (~+45% relative, sub-millisecond absolute)** — negligible next to any embedding/LLM reranker (100–1000 ms + $), so fusion stays firmly in the deterministic/cheap regime. (The scale bench also caught two measurement traps, now documented in it: identical flood distractors are collapsed by context-dedup so collisions must use *distinct* documents; and query-derived answer keys leak the query term into the weighted FTS `key` column and mask the effect — real keys are content-hashes, so the bench uses opaque keys.) **Reaches the production integrated path:** fusion lives in the store's `fts_search`, which BOTH `recall_topk_fts` (direct) and `cascade_retrieve`'s FTS supplement route through — so `recall_cascade` (the pipeline Permagent uses) benefits transparently. Verified in `fusion_scale_bench`: the integrated cascade path shows the same lift as the direct path (collision 0.14→1.00, inflection 0.83→1.00, ALL recall@5 0.70→1.00), because short keyword queries underfill TACT (fingerprint/wing matching is weak on bare keywords) and the fused FTS supplement carries the recall. Promotion to default is gated on a real-corpus (LongMemEval-S) confirmation, not yet run on this machine (dataset unavailable here).

### Added (feat/anticipatory-in-recall)
- **In-recall anticipatory augmentation** (`SPECTRAL_ANTICIPATORY_RECALL=1`, off by default). Folds the lift recommender (`Brain::recommend`) directly into `recall_topk_fts` so a consumer gets anticipatory miss-recovery for free — no manual `recall + recommend` composition. After the query-matched results (and after retrieval-event logging, so anticipated memories are **not** logged as retrieved — avoids a recommend→log→recommend feedback runaway), recall seeds from its top hits, pulls their lift-associated memories (`lift ≥ 1.0`, above-baseline association only), and appends up to 3 that the query did not already surface. Anticipated hits are **visibility-filtered** like everything else and carry `hits = 0` so a consumer can distinguish association-surfaced from keyword-matched. Verified end-to-end (`anticipatory_bench` + `brain_tests::anticipatory_recall_augments_query_miss_when_enabled`): a "kubernetes rollout strategy" query that keyword-matches only the deploy memory returns `["kube-deploy"]` OFF and `["kube-deploy", "deploy-outage"]` ON — recall itself recovers the postmortem the query missed, ranked by context-specific lift, not popularity.

### Fixed (feat/recency-consistency)
- **Recency ranking fixed: consistent across native/imported memories, and no longer able to bury a relevant old answer.** Two coupled defects on the default recall path. (1) **Silently inert for imports:** the recency parser accepted only the SQLite `datetime` format, but `RememberOpts.created_at` (historical imports — dated conversation history) is stored as RFC3339 (`ingest.rs`), so the parse failed → `age=0` → no decay. Recency worked for natively-created memories but not imports — inconsistent by origin, and invisible in every prior bench because freshly-created memories are all age≈0. New `parse_created_at` accepts both formats. (2) **Too aggressive:** recency was a **multiplicative** decay (`score *= 0.5^(age/365)`) that could drive an old-but-relevant answer's composite score toward zero and demote it out of top-K — a latent recall regression for exactly LongMemEval's multi-session/temporal answers and Permagent's imported history. It is now a **bounded additive boost** (`score += RECENCY_BOOST_WEIGHT · freshness`, weight 0.1), consistent with the declarative/entity boosts: a gentle tiebreaker that lifts fresh content among near-equal-relevance neighbors but cannot override a real relevance gap. Verified deterministically (`spectral-bench-real --bin recency_probe`): recency is now active for RFC3339 imports (ON rank 6 ≠ OFF rank 1) **and** a 10-year-old relevant answer stays in top-40 (the old multiplier would have annihilated it to ~rank 51). `ranking::recency_uses_context_now_not_utc_now` rewritten for the bounded semantics; full graph + ingest suites green. Found by auditing the one default-path signal the recall-expansion sweep hadn't examined.

### Added (feat/query-aliases)
- **Curated query alias/synonym expansion** (`SPECTRAL_QUERY_ALIASES=/path/to/aliases.json`, unset by default). The deterministic answer to the semantic-bridging gap pure lexical matching cannot close (`CEO`↔`chief executive`, `k8s`↔`kubernetes`): `fts_query_words` expands each query term matching an alias key with that alias's terms (multi-word expansions re-tokenized into individual FTS terms). Spectral supplies the **mechanism**; the consumer supplies the **controlled vocabulary** — which the near-duplicate literature endorses as the *one* safe synonym approach (a curated domain table, never a general thesaurus, which harms precision). The table is loaded once and cached (`OnceLock`); an unset/unreadable var yields an empty table (a strict no-op — proven by `query_aliases_empty_by_default`). Additive recall expansion; the re-ranker handles precision. Unit-tested for case-insensitive keys, multi-word tokenization, and empty-table no-op; end-to-end validated in `spectral-bench-real --bin recall_expansion_bench` (`chief executive raise` retrieves a `CEO …` memory sharing no other token). This closes the deterministic recall-expansion frontier: the remaining gaps are write-side identifier tokenization (camelCase — deferred, needs a standalone split index, chat-irrelevant) and true paraphrase (unreachable without embeddings).

### Added (feat/number-word-bridging)
- **Number-word bridging** (`SPECTRAL_NUMBER_NORMALIZE=1`, off by default). Query `3 puppies` and content `three golden retrievers` share no matchable token, so the answer is absent from the pool — a recall gap in LongMemEval's counting/number category. With the flag on, `fts_query_words` ensures both surface forms of a number are MATCH terms (`3` ⇄ `three`). It scans the **raw** query so a single digit (dropped by the `len > 1` filter before it could bridge) still counts. The map is a **closed, unambiguous** set — cardinals 2–12 only; `one`/`zero` and homographs like `second`/`quarter`/`half` are deliberately excluded (the same discipline as the stopword set), because number words are *universal English*, not a domain vocabulary — the one semantic bridge safe to add where a general synonym table (which the near-dup literature flags as precision-harming) would not be. Additive recall expansion; the re-ranker handles precision. Verified deterministically (`tokenization_probe`: OFF misses `3 puppies`→`three…`, ON retrieves it); unit-tested for both directions and homograph exclusion. Opt-in pending real-corpus measurement.

### Fixed (feat/query-separator-tokenization)
- **Query separators are now SPLIT, not deleted — fixes a whole class of "answer never in the pool" recall failures.** `fts_query_words` filtered out non-word characters, which *merged* adjacent tokens into a string that matches nothing the way FTS5 tokenized the stored content: `alice@acme.io` → `aliceacmeio`, `and/or` → `andor`, `api.acme.dev` → `apiacmedev`. The answer memory never entered the candidate pool at all (not a ranking problem — a **recall** problem, the only kind that moves recall@K, LongMemEval's operating point). The sanitizer now **splits** on every character unicode61 treats as a separator (everything except alphanumerics and intra-token `-`/`_`/apostrophe), tokenizing the query the same way the content was tokenized. This subsumes the earlier possessive special-case (`Marcus's` → `Marcus`) and is a genuine recall **expansion** (moves answers from absent → present), unlike rerank-level reordering. Found via a systematic tokenization sweep (`spectral-bench-real --bin tokenization_probe`) that also flagged a remaining, higher-cost class — **camelCase content** (`taskRunner` isn't found by `task runner`) needing write-side identifier splitting — logged as a future lever (less relevant to chat-style corpora). Full graph + ingest suites green; new unit test `splits_on_separators_instead_of_merging`.

### Fixed (feat/possessive-recall)
- **Possessive-entity queries no longer silently drop the answer memory.** `recall_topk_fts` sanitized query terms by keeping word characters, so a possessive like `Marcus's` became `Marcuss` (apostrophe dropped, trailing `s` kept) — which porter-stems differently than the entity `Marcus` in stored content and never matched, dropping the answer-bearing memory from the candidate pool entirely (not a ranking issue — it never entered the pool). Any `X's title / Sarah's job / the company's policy` question was affected. New shared `fts_query_words` helper strips a trailing possessive `'s`/`’s` before tokenizing, used by both FTS sanitization sites. Root-caused via deterministic repro (miss → hit) and validated end-to-end: a question that returned "I don't know" now retrieves the fact and answers correctly. (TACT's regex path already split on the apostrophe and was unaffected.)

### Added (feat/ambient-feedback)
- **Ambient recurrence feedback (write-time importance signal).** When new content re-encounters an existing memory — detected by the recognition engine (not exact hash; catches restatements that reuse the salient specifics) — the prior memory is reinforced. Importance emerges from *recurrence* (content-driven), the opposite of the retrieval-popularity signal that failed as co-retrieval. `RememberResult.recurrence` surfaces the matched prior + familiarity so a consumer can consolidate the near-duplicate (fewer memories → cheaper). Enabled via `SPECTRAL_RECURRENCE_FEEDBACK=1` (off by default, measure-first). Verified deterministic ($0, no LLM): OFF = no effect; ON = re-encounter detected (familiarity 0.96) and prior strengthened, once-mentioned facts untouched (`recurrence_bench`).
- **Anticipatory recall via lift (`Brain::recommend`).** A recommender over the user's *own* memories: given a seed, surface memories ranked by **lift** (`co_count · total / (occ(seed)·occ(this))`) — context-specific association — instead of raw co-retrieval count. Lift is the association measure recommender systems use to avoid pushing globally-popular items to everyone; it is the deterministic, zero-LLM **fix for the popularity bias that sank raw co-retrieval**. Unit-proven: on a scenario where a globally-popular memory co-occurs with everything, raw `co_count` ranks it top for every seed while lift correctly ranks the seed-*specific* memory first. New `MemoryStore::recommend_by_lift`, `RelatedMemory.lift`. Demonstrated end-to-end (`anticipatory_bench`) through the real recall→retrieval_events→rebuild→recommend loop: a query that FTS-misses an associated memory (shares no query terms) still surfaces it via `recommend(top hit)` — "what you need before you ask" — with the popular blob suppressed.

### Changed (feat/rrf-fusion)
- **Federated merge now uses Reciprocal Rank Fusion (RRF)** — the field-standard rank-fusion method (Cormack et al. 2009), replacing the custom rank-normalization + corroboration boost. RRF subsumes both guarantees in one widely-accepted primitive: it ranks on within-child *position* (immune to self-asserted score inflation → collapses poisoning floods) and sums contributions across *distinct* members (genuine cross-member corroboration outranks a lone assertion). Contributions are deduped per origin, so a member cannot self-corroborate by flooding identical copies. `MergePolicy` is now `{ fusion: FusionMethod, per_child_cap }` with `FusionMethod::{Rrf { k }, RawScore}` (default `Rrf { k: 60 }`); signed provenance and the per-child cap remain the differentiated layer on top.

### Added (feat/descriptions-in-actor-context)
- **Librarian descriptions can now be surfaced to the actor** (`SPECTRAL_ACTOR_DESCRIPTIONS=1`, off by default). Until now a memory's `description` (written by Permagent's Librarian) was a retrieval-only signal (FTS column, item #8, +2.5pp) invisible to the actor — the context formatter rendered `content` only. With the flag on, each turn in the actor context gains a labeled `[librarian: …]` gloss. Clean on/off ablation: apply the same descriptions in both arms (constant FTS effect), toggle only whether the actor sees them, to isolate the SYNTHESIS effect on the categories FTS enrichment alone could not move (multi-session / temporal). New `desc_context` bin measures the mechanism + token cost locally. **Measured (10-question difficulty-engineered micro-eval, Sonnet 4.6, both arms): +0pp accuracy (9/10 → 9/10), +7% context tokens / +11% cost — a negative result.** Failure analysis: descriptions in the actor context are redundant when the memory is retrieved (content already states the fact) and useless when it isn't (the one failure was a retrieval miss with no gloss in context); descriptions earn their keep at *retrieval* time, not synthesis time. Disposition: **flag stays OFF by default**; keep descriptions retrieval-side. See docs/internal/descriptions-in-actor-context-2026-07-11.md.

### Added (feat/minhash-recognition)
- **Shingle-set (MinHash) recognition channel** — a widely-accepted lexical near-duplicate technique added alongside the peak-pair and winnowed-gram channels, integrated to lift recognition on its strong regime (re-encounter detection) while keeping the differentiated auditable verdict/evidence layer. New `spectral_recognition::minhash` (token k-shingles, **containment** scoring `|probe ∩ doc| / |probe|` — the fragment-appropriate similarity, plus MinHash signature/LSH banding for scale). `RecognitionConfig::minhash` (`weight = 0.0` disables); the engine enrolls each memory's shingle set into an inverted shingle index and fuses containment into the score, evidence (`minhash: containment 0.87`), and the familiarity scalar. `RecognitionStore` gained `index_minhash` / `lookup_minhash` (SQLite + in-memory), and `forget_memory` now purges the MinHash tables too. Measured lift (`recognition_bench`): on a hard corpus with lexical near-miss negatives and heavily-degraded (72%) positives, familiar-vs-near-miss **ROC-AUC 0.880 → 0.990 (+0.109)** and **degraded re-encounters recovered 9/12 → 12/12**.

### Added (feat/benchmarks)
- **Federation poisoning-resistance benchmark** (`spectral-bench-real --bin poison_bench`) reporting Attack Success Rate (ASR), the accepted metric from the 2026 memory-poisoning literature (MPBench, AgentPoison). On an 8-question shared-project-brain with two honest members and one attacker flooding 6× max-signal poisons per question, ASR drops from **87.5% (undefended raw-score merge) to 0.0% (shipped RRF default)** — an 87.5pp reduction. (This benchmark surfaced and fixed the RRF self-corroboration bug during development.)
- **Recognition re-encounter benchmark** (`spectral-bench-real --bin recognition_bench`) reporting familiar-vs-novel ROC-AUC on the recognition primitive's actual job (re-encounter detection, exact + degraded). On a clean ops corpus: AUC 1.000, all 40%-degraded re-encounters still Familiar, all novels correctly flagged Novel — deterministic, embedding-free, evidence-bearing. (Honest scope: small in-domain corpus; the at-scale lexical AUC ~0.94 and the MinHash comparison are in RECOGNITION_BASELINE.md; semantic paraphrase is not claimed.)

### Added (feat/signed-contributions)
- **Signed memory provenance (federation roadmap §3.1).** Every memory written through `Brain::remember*` is now signed with the brain's Ed25519 identity: the row carries `source_brain_id` (the authoring `BrainId`) and a `signature` over `(source_brain_id, content_hash, created_at, visibility)`. New `spectral_core::identity`: `memory_signing_payload` (domain-separated, length-prefixed canonical bytes), `BrainIdentity::sign_memory`, and `verify_memory_signature`. `MemoryHit` and `Memory` gained `source_brain_id` + `signature` (schema migration adds the columns, NULL on legacy rows); `MemoryStore::set_signature` stamps them post-write over the stored values. `Brain::verify_hit(hit, pubkey)` (and the umbrella wrapper) authenticate a hit — `true` only if it is signed, the key matches its `source_brain_id`, and the signature covers its content, timestamp, and visibility; tampering with content or escalating visibility fails verification, and a foreign key cannot impersonate the origin. `Brain::verifying_key()` exposes a brain's public key for peers. This is the trust anchor for multi-contributor project brains — attribution and anti-poisoning for a determined (signed) insider, complementing the coordinator-side merge trust below.

### Added (feat/federation-trust-and-forget)
- **`Brain::forget(key)` — verified hard delete / right-to-be-forgotten.** Unlike `consolidate_into` (soft hide from two read paths), this purges every SQLite substrate a memory touches (row, FTS shadow, constellation fingerprints, spectrogram, annotations, consolidation edges, co-retrieval pairs, retrieval-event references) **and** the recognition sidecar's pair/gram index (a `forget_memory` was added to `RecognitionStore` — no unenroll existed before), then re-probes recall and recognition and returns a `ForgetReport` with per-substrate counts and `fully_forgotten()`. New `MemoryStore::delete_memory_by_key` returns a `ForgetReceipt`; `RecognitionEngine::forget` / `RecognitionStore::forget_memory` added.
- **Trust-aware federation merge (poisoning defense).** `FederationCoordinator::fan_out_recall_with_policy` + a `MergePolicy` control how member scores are combined. Defaults are safe: **rank-normalization** (each child's raw `signal_score` is replaced by a within-child rank ramp, so a member cannot dominate the global merge by self-asserting max scores — it collapses a max-signal flood from a wall-at-the-top to a decaying ramp) and **corroboration boost** (content independently contributed by K members outranks an uncorroborated memory; a member cannot self-corroborate). Optional `per_child_cap` bounds any one member's contribution. `MergePolicy::raw_scores()` reproduces the legacy behavior. The merge logic is a pure, unit-tested `merge_and_rank` function. Grounded in the verified 2026 memory-poisoning literature (see docs/internal/federation-fundamentals-2026-07-10.md); does not stop a determined signed insider — that needs the signed-provenance + reputation layer (next).

### Added (feat/federation-fundamentals)
- **Read-only brain mode** (`BrainConfig::read_only`, `BrainBuilder::read_only()`, `SqliteStoreConfig::read_only`). Opening never mutates the brain: no directory/identity creation, no schema DDL, no migrations, no FTS tokenizer rebuild, no fingerprint backfill; SQLite and the recognition sidecar open with `SQLITE_OPEN_READ_ONLY` + `PRAGMA query_only`, Kuzu opens with engine-level `read_only`. Recall paths skip their ambient writes (cascade auto-reinforce, retrieval-event logging), and 19 write APIs return the new `Error::ReadOnly` instead of mutating. This is the required mode for federated read-time fan-out over a brain you don't own — previously, merely opening a peer's brain ran migrations against it (including a whole-index FTS rebuild if tokenizer configs differed), and every recall inflated its signal scores and logged the caller's query metadata into its store.
- **Federation visibility boundary**: `FederationCoordinator::fan_out_recall` now takes a `Visibility` context and filters each child's hits before merging (`content >= context`, same rule as single-brain recall). Previously the fan-out path applied **no** visibility filtering — children's `Private` memories crossed the federation boundary. Multi-user coordinators pass `Visibility::Team`+; single-user coordinators over their own brains pass `Visibility::Private`. (Signature change; labels are self-asserted by the data owner, so this is honest-participant privacy, not mandatory access control.)

### Fixed (feat/federation-fundamentals)
- `FederationCoordinator::add_brain_weighted` no longer double-counts a re-added `BrainId`: re-adding replaces the existing child handle (mirroring the registry's replace-on-duplicate semantics) instead of pushing a duplicate child.
- Spectrogram wing-corpus construction now uses the bounded, LRU-cached `wing_search` (top-256 by signal) instead of materializing the entire wing per `remember()` when spectrograms are enabled.

### Fixed (feat/porter-default)
- **Spectrogram novelty dimension no longer constant 1.0.** The analyzer previously always received an empty wing corpus (`AnalysisContext::default()`), so novelty was 1.0 for every memory (measured in docs/internal/SPECTROGRAM_AUDIT.md: 497/497). `remember_with`, `backfill_spectrograms`, `audit_spectrogram`, and `recall_cross_wing` now build the corpus from other same-wing memories (capped at 64 KB); a near-duplicate of existing wing content scores low novelty, genuinely new content scores high.
- **`wing_search` now honors its query terms.** TACT Tier-2 wing retrieval was a signal-ordered wing dump that ignored the query (`_query_terms`). It now stably re-ranks the wing set by distinct-query-term overlap (signal order preserved within ties, no memory dropped), including on the LRU-cached path — matching the behavior its trait documentation always promised.

### Performance
- `SqliteStore` now memory-maps the database file via adaptive `mmap_size` PRAGMA (50 MB – 1 GB based on file size). This eliminates p99 latency spikes caused by SQLite page cache eviction on databases larger than ~10 MB. On the migrated-brain reference benchmark (1000 iterations): aggregate warm p95 improved 9% (2.35 → 2.14 ms), p99 improved 17% (2.65 → 2.20 ms), worst per-query p99 dropped from 17.18 ms to 7.41 ms, and queries with p99/p95 > 2.0x went from 9 to 2. Configurable via `SqliteStoreConfig::mmap_size` (or `BrainConfig::sqlite_mmap_size`).

### Performance
- Wing result LRU cache (32 entries) in `SqliteStore` — serves repeated `wing_search()` from memory, invalidated on `write()`
- Compound `(wing, anchor_hall)` and `(wing, target_hall)` indexes on `constellation_fingerprints` — accelerates hall-match path

### Notes
- These additions are non-breaking. Existing code calling `Brain::remember(key, content, visibility)` continues to work; new fields default to `None`/`1.0`.
- `MemoryStore::write()` trait signature unchanged — the new fields are on the `Memory` struct it already accepts.

- **`Brain::recall()`** now takes `context_visibility: Visibility` to filter
